DELIVERABLE CONTAINING PROCESS OF THE USER TESTING PLAN FOR THE GROUP PROJECT.

INTRO

This deliverable page presents a summary of the user research plan which has been conducted with users of the mobile app of Grip on Sound. The page gives a summary of the process which has been taken, from setting up goals to execution. It is suggested to read through this summary first for a better understanding, after which to use the button at the bottom of the page to refer to as the actual proof document. The linked research question to this deliverable is:

  • How can we test the UI/UX of the app?

CONTEXT

After creating a design for the mobile app, it is important to validate the concept. The approach to this validation will be discussed in this deliverable. The design is an iterative process, which means that during all times designs could be tested and validated. This will lead to several testing sessions, spread out over a longer period of time.

For me this meant that I had started creating this user testing plan at an early stage of the project, when the design was not even started.

This meant that a barebones setup for the user testing plan was already there for the group, and after having decided on what to test, this would be added.

GOALS

The eventual goal of the user testing plan was to validate the concept. I have created several goals as SMART as possible to achieve this:

  • We want to understand why a user within our target group takes certain steps to fulfill a task in our app, with the help of different testing methods, in order to understand which aspects of the usability need improvement, in sessions which will take roughly half an hour.
  • We want to document the flaws and pain points of our app, by using different tools, from which we can draw conclusions. This will be done during and after the research sessions.
  • We want to get a better understanding of our target audience, to clarify our target audience, by interacting with the user during our research sessions of approximately half an hour.
  • We want to understand whether we should use a one-page application or a multiple page application, by making use of several comparison testing methods, during the interactions we have with the user during our testing sessions, which will take approximately half an hour.

METHODOLOGY

After having done research on specific research methods, and knowing what needed to be tested and validated in the app, a final decision on the testing method was made. Firstly, I will show what the requirements for the validation are:

  • The homepage for the daily dose needs to be validated.
  • The amount and type of data on the insights need to be validated.
  • The calendar page overview needs to be validated.
  • The information on the content page needs to be clear.
  • The user flow and navigation should be clear and easy to use.

After having everything prepared, the decision was made to make use of two research methods:

  • Moderated User Testing.
  • Unmoderated User Testing.

This research document mainly focuses on the moderated user testing I have performed with several group members and its results, while the unmoderated user testing has been taken care of by other group members. The way of conducting moderated user testing is done individually between the researcher(s) and the participant.

EVALUATION & NEXT STEPS

After having conducted the user testing plan with several users, multiple results came forward, which lead to a list of action points for a next iteration. This action list is as follows:

  • Changes on context on the pages.
  • Making sure the colors do not contrast and cause for confusion
  • Merge the insights page with the homepage, because of unclarity about the calendar page.
  • Change the order of the months on the month overview.
  • Changes to the navbar for the active pages.
