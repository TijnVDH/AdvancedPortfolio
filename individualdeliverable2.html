WIREFRAMES AND MOCKUPS

DELIVERABLE CONTAINING DOCUMENTING THE PROCESS OF ITERATIVELY WORKING ON THE DESIGN OF THE PORTFOLIO.

INTRO

This deliverable page presents the documentation of the wireframing and designing process of the portfolio. After having answered several sub-questions, it was time to start structuring the portfolio. I started doing this by creating wireframes in a quick succession. The linked research questions to this deliverable are:

  • Hoe wil ik mezelf presenteren als media designer in mijn portfolio?
  • Hoe ga ik mijn expertise in media design duidelijk naar voren brengen in mijn portfolio?
  • Hoe kan ik een gebruiksvriendelijke portfolio opstellen, waardoor de gebruiker niet zelf naar informatie hoeft te zoeken?

WIREFRAMES

After gathering a solid idea on what the portfolio should contain by consulting the canvas course, and talking with lecturers, I was able to set up the list of requirements:

  • The portfolio should contain a seperately delivered reading guide.
  • The portfolio should contain a clear flow of: reading guide -> deliverables -> documentation.
  • The portfolio should indicate proficiency in all of the learning outcomes.
  • The portfolio should contain all different projects that have been worked on during the semester.

By setting up these requirements, I started ticking them off in my initial wireframes. The homepage would contain a link to all three projects for ease of accessibility. Within these projects, deliverables for that specific project are shown. On this page, the user would also be able to toggle between the projects in the top right.

Before accessing a deliverable, there are tags referring to each deliverable indicating the learning outcome it is attached to. Within a deliverable, an overview of the proof is shown, and when necessary, external documents are linked at the bottom of the deliverable.

These wireframes were very low fidelity, but after making use of peer review I could validate that the structure of the portfolio was clear. This meant that I could quickly start with the actual design of the portfolio.

MOCKUPS

The start of the actual design was a bit rough. Making design choices such as colors, typefaces and spacing took longer than expected, which was not particularly a bad thing, but I wish I could have went through this in a more fluent succession. On the other hand, since it took longer for me to make certain choices, I dedicated more time to asking for feedback on certain aspects of the portfolio. This helped with creating a portfolio design I was sufficed with.

I also wanted my design to be in a sleek dark mode. This is a decision I had made since this is how I usually create my sites. All projects would eventually have a clear illustration, indicating the project it refers to (the group project will have the grip on sound logo for example).

PROTOTYPE & ITERATIONS

After finishing my design, I turned my mockups into a functional prototype. This is the prototype I used to execute my user research plan. This prototype had the main functionalities of the prototype working. This includes:

  • A functional explore button which scrolls the user down.
  • An accessible backwards button at all times.
  • A selection of three projects for the user to choose between.
  • The ability to toggle between projects on a specific project page.
  • The ability to read through deliverables of a specific project.

After having executed this user research plan, I noticed that the overall structure is clear, but the portfolio was still missing some features. I decided to create a diagram of the final structure. This structure still has the same idea, but with added features.

These features consisted mainly of a main reading guide with a competence table on the homepage. This decision was made for the ease of access to solid proof. In this table the user is able to see all of the deliverables of the semester which are linked to a specific learning outcome. This meant that the deliverbales are accessible through the competence table on the homepage, and on the project pages.

COMPETENCE TABLE

Adding these additional features, meant that these also needed to be worked out. Something I realized during update meetings, is that the competence table really acts as the core of the portfolio, where the proof is gathered. This meant that I took some time to work out the design of the competence table. The choice of a simple table was easily made, since this is the part where all learning outcomes are listed, with their connected proof, which can be seen down below.

NEXT STEPS

With my designs ready, I am ready to set up my user research plan. This research plan will help me to iterate on my portfolio, and most importantly validate the design I have created. If any improvements are the results of this user research plan, I will iteratevily rework the portfolio. This caused the list of action points for the following of the project:

  • Validate the created prototype with the user research plan.
  • Iterate on the portfolio if needed before jumping in development.
